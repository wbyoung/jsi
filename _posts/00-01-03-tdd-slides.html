Test Driven Development

What is it?

Let's break it down.

What are tests?

Tests verify that your code works properly.

How do you create tests?

Well, you write more code.

A function to test


/**
 * Calculate the difference between two numbers
 * @param {number} a - The first number
 * @param {number} b - The second number
 * @returns {number} The difference between a and b
 */
var difference = function(a, b) {
  // calculate difference here
};

What should we test?

  • Standard uses
  • Corner cases (unusual uses)
Corner cases are sometimes difficult to think of as they occur using unusual or unexpected input. What might a corner case be for a square-root function? Negative numbers.

A test


var result = difference(5, 2);
expect(result).to.eql(3);

What do you think this does?

Tools

Mocha and Chai

Programmers have run out of descriptive names.

Chai.js gives us natural language to express expectations.


var result = difference(5, 2);
expect(result).to.eql(3);

Expressing an expectation is called an assertion.

Mocha allows us to group tests


describe('difference()', function() {
  it('returns a positive number when the first argument is greater', function() {
    var result = difference(5, 2);
    expect(result).to.eql(3);
  });
  it('returns a positive number when the second argument is greater', function() {
    var result = difference(2, 4);
    expect(result).to.eql(2);
  });
});

You'll see shortly how this all ties together nicely.

Driving Development

Remember, it's called Test Driven Development.

Workflow

  • Write a test
  • Run test and make sure it fails
  • Write the actual code
  • Run test again to see that it passes
  • Repeat
This process is known as the red/green/refactor cycle.

Buzzwords

  • Unit Tests
  • Functional Tests
  • Integration Tests
  • End to End Tests
  • Acceptance Tests
  • BDD, Behavior Driven Development
  • Agile development

The lines between different testing terms can be fuzzy. Some people are very passionate about the differences.

  • Unit - typically tests a single function, simple
  • Functional - solely checks that result is correct, state/side effects not considered
  • Integration - tests systems together, could include db/disk
  • Acceptance - plain English tests, describes using full system
  • BDD - focus on behavior, the conversation around behavior, and terminology like given, should and scenarios
  • Agile - most shops do something like this, includes TDD, short iteration cycles

There's a good answer on StackOverflow about the difference between some of these terms.
